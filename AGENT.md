### 1. 商店鎖定與物品合成機制問題
* **遭遇問題**：玩家點擊鎖定背包物品（左鍵點擊鎖定不參與合成）生效，但右鍵點擊鎖定商店商品時，在進入下一波或手動刷新時商店商品依舊會被刷掉，鎖定無效。
* **原因分析**：商店刷新（Refresh）邏輯在更新 `CurrentShopItems` 時，未檢驗對應欄位的 `ShopSlotsLocked` 狀態，導致強制覆蓋。
* **解決方案**：
  * 修改商店刷新邏輯，每次在產生新商品時，優先判斷 `ShopSlotsLocked[i]` 是否為 `true`。
  * 若被鎖定，則跳過該欄位的更新與扣錢，原商品及其狀態原封不動保留。

### 2. 解析度適配與鏡頭平滑跟隨機制
* **遭遇問題**：
  1. 遊戲解析度需要調整為 1600*1200 比例，並能隨時切換大、小與全螢幕。
  2. 原本鏡頭死板跟隨角色，玩家希望歐克在畫面中央能自由活動，當靠近畫面兩側邊緣時，相機鏡頭才向外移動。
* **解決方案**：
  * **解析度適配**：在 `HUD.gd` 中監聽 F1、F2、F3 快速鍵，呼叫 `DisplayServer.window_set_size()`，分別將視窗大小設定為 800*600、1600*1200，並透過 F3/F11 切換全螢幕（`DisplayServer.WINDOW_MODE_EXCLUSIVE_FULLSCREEN`）。
  * **視差邊界相機跟隨**：修改 `Camera2D` 的跟隨邏輯。不採用硬綁定子節點，而是在每一幀計算主角相對於相機視口中心的水平位移。若位移小於緩衝半寬（Threshold，如 200 像素），相機保持不動；若超出此半寬，相機向該方向平滑移動（`lerp` 差值插補），既保證了操作的流暢感，又避免了畫面頻繁晃動。

### 3. GDScript `queue_free()` 連續釋放崩潰 (Double-Free)
* **遭遇問題**：
  ```
  E 0:01:55:253   BackpackUI._rebuild_placed_items: Attempt to call function 'queue_free' in base 'previously freed' on a null instance.
  ```
* **原因分析**：在快速拖放物品、合成、刷新商店或切換載體時，UI 節點可能在當前幀被引擎或其他信號回呼（如 `drag_data` 銷毀）先行釋放，但在重建 UI 時遍歷字典重複呼叫了 `queue_free()`。
* **解決方案**：
  * 對所有批次銷毀 UI 節點的 `queue_free()` 呼叫，在執行前一律包裹 **`is_instance_valid(node)`** 檢查。
  * 確保只有當節點確實存在且尚未被釋放時，才將其加入銷毀隊列。

### 4. 主角與敵人體型放大與跳躍物理調整
* **遭遇問題**：
  1. 原版角色體型過小，根本看不到 Vessel（載體）與 Spirit 角色的狀態和動作。
  2. 放大體型後跳躍物理過輕，或是主角能夠輕易越過 192 像素高的怪物。
* **解決方案**：
  * **比例調整**：設定主角身高為 256 像素，將 Vessel 及怪物的體型同步等比放大（保持主角大約是敵人和載體的 1.7 倍高，例如怪物放大為 `75 * 150` 像素）。
  * **跳躍重力修正**：將跳躍初速度調整為合理區間（例如 `-550.0`），配合大體型下的重力加速度，確保跳躍高度合理，並且**無法完全跳過敵人**以維持戰鬥阻擋體驗。

### 5. 敵人接觸傷害改為多技能輪換冷卻機制
* **遭遇問題**：敵人原本只要碰到主角就造成每秒高額接觸傷害，戰鬥體驗單調且主角極易暴斃。
* **原因分析**：缺乏攻擊前搖和戰鬥招式冷卻設計。
* **解決方案**：
  * 移除怪物的 Body Contact 扣血，改為基於範圍與冷卻的攻擊系統。
  * 為敵人實作技能輪換清單（如 `Claw Slash` 與 `Heavy Cleave`），當進入攻擊範圍時，敵人停止移動，並觸發短暫的閃白 Telegraph (前搖預警)，隨後造成判定範圍內的傷害，之後進入對應冷卻。

### 6. GDScript 命名衝突遮蔽全域內建函數
* **遭遇問題**：
  ```
  W 0:00:17:313   GDScript::reload: The variable "range" has the same name as a built-in function.
  ```
* **原因分析**：在 `EnemyBase.gd` 中定義了成員變數 `var range`，遮蔽了 GDScript 的全域內建函數 `range()`。
* **解決方案**：
  * 將變數名稱重構為更具語意且不衝突的變數名，如 `action_range`。

### 7. C# 與 GDScript 屬性重命名不同步崩潰
* **遭遇問題**：
  ```
  E 0:00:17:359   HUD._update_pip_expression: Invalid access to property or key 'CurrentFemale' on a base object of type 'Node (InventoryManager.cs)'.
  ```
* **原因分析**：在將載體的英文名稱由 `Female` 統一更名為 `Vessel` 時，雖然 C# 部分全部成功更名，但忘記了 GDScript 中的 `HUD.gd` 與 `Player.gd` 等檔案依然在讀取 `InventoryManager.CurrentFemale` 等舊屬性，導致執行期因找不到屬性而閃退。
* **解決方案**：
  * 透過全域搜尋並將所有 `Female` 相關命名統一修改為 `Vessel`。
  * **經驗教訓**：在進行跨語言 (C# & GDScript) 屬性或信號變更時，必須先梳理依賴關係（建議善用 `/graphify` 分析鏈路），確保兩端在同一批次內同步重構。

### 8. 有效格與無效(過渡)格機制及屬性比例縮放
* **遭遇問題**：不同角色網格排布不同，道具為了要滿足部位限制，又不能互相碰撞卡死，需要一種「過渡」的網格設計。此外，道具效果需要根據實際落在有效區域的比例來計算。
* **解決方案**：
  * **無效格 (Inactive Zone) 設計**：在 `PlacementZone` 增加 `Inactive` 類型。在 `GridData.CanPlaceItem()` 與放置/移除機制中，如果格子屬於 `Inactive`，則**不檢測碰撞，亦不標記佔用**。這使得多個道具的邊緣或手柄可以疊加放置在無效格上。
  * **比例計算**：在 `GridItemInstance` 中實作 `GetActiveRatio()`，計算該道具覆蓋在「非無效格」的單元佔總體單元的比例。
  * **數值縮放**：在 `CombatManager.cs` 中，道具的屬性加成、武器最終傷害皆乘以該比例；而武器冷卻時間則除以該比例（比例越低，冷卻時間越長，射速變慢）。

### 9. 載體 (Vessel) 獨立背包裝備狀態與保留切換機制
* **遭遇問題**：原本 `InventoryManager` 只有一個全域共享的 `BackpackGrid`。當切換 Vessel 時，若勾選「保留裝備」，會強行嘗試在新 Vessel 上重新擺放（因為結構不同，大部分道具會被迫卸下回 Stash）；若未勾選則全部卸下。
* **原因分析**：各個角色沒有自己獨立的裝備背包數據，導致無法實現「法師滿裝、切換弓手為空，但法師裝備依舊綁定法師」的保留機制。
* **解決方案**：
  * **字典映射獨立網格**：在 `InventoryManager.cs` 中宣告 `Dictionary<string, GridData> VesselGrids`，使每個載體 ID 對應一個獨立的 `GridData`。
  * **動態 Getter**：將 `BackpackGrid` 改寫為動態屬性，每次讀取時根據當前活躍的 `CurrentVessel` 從字典中提取對應網格。這使得外部其他模組完全不需修改，即可無縫適配多角色獨立背包。
  * **切換邏輯**：
    * `keepEquipped = false`：先清空原 Vessel 的專屬網格，將裝備退回 `Stash`，然後切換角色。
    * `keepEquipped = true`：不變動原角色網格，直接切換角色，新角色顯示新角色網格的裝備。

### 10. 切換載體時 ItemUI 視覺元件殘留 Bug
* **遭遇問題**：在勾選保留裝備切換角色時，後台數據正常，新角色網格為空，但畫面上舊角色的道具仍停留在原位，沒有消失。
* **原因分析**：`BackpackUI.gd` 原本的 `_rebuild_placed_items()` 是透過遍歷 `item_nodes` 的 keys 來銷毀舊 `ItemUI` 節點。在切換角色時，因為數據解綁等生命週期斷檔，字典內的 reference 與畫面上實際子節點不一致，造成部分 `ItemUI` 漏刪而殘留在 `grid_container` 內。
* **解決方案**：
  * 拋棄依賴舊字典 key 的清除方式。
  * 重構為直接遍歷 `grid_container` 的所有子節點，凡是 `child is ItemUI` 的節點一律安全調用 `queue_free()` 予以刪除。這 100% 確保了在角色切換時，畫面上的道具元件會被徹底清理乾淨。
